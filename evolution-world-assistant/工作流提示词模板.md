# Evolution World — 工作流提示词配置指南

> 本文档面向**工作流配置者**（用户），帮助你理解如何编写工作流提示词，让 AI 正确输出 JSON 来更新世界书。

---

## 一、工作流输出的 JSON 是什么？

每次工作流执行时，EW 插件会用你编排好的提示词去请求一个 AI 模型。这个 AI 需要返回一个 JSON 对象，告诉插件：

1. **要写入哪些条目**（角色档案、记忆、时间线等）
2. **要加载哪些条目到对话上下文**（让主聊天 AI 能看到）

就这两件事。

---

## 二、JSON 格式：必须有什么？可以没有什么？

### ✅ 必须有

| 字段 | 说明 |
|---|---|
| `operations.worldbook.desired_entries` | 你要创建或更新的条目列表 |
| `operations.controller_model` | 控制哪些条目会被加载到对话中 |

### ⚠️ 推荐有

| 字段 | 说明 |
|---|---|
| `operations.worldbook.remove_entries` | 要删除的条目（没有就写空数组 `[]`） |

### ❌ 不需要写（插件自动处理）

| 字段 | 插件怎么处理的 |
|---|---|
| `version` | 自动补全为 `"ew-flow/v1"` |
| `flow_id` | 自动用工作流 ID 填充 |
| `status` | 自动设为 `"ok"` |
| `priority` | 自动用工作流配置的优先级 |
| `enabled` | Dyn 条目自动设为关闭（由控制器按需加载） |
| `diagnostics` | 自动生成 |
| 条目名的 `EW/Dyn/` 前缀 | 自动添加，你只需要写裸名 |

### 最简 JSON 长这样

```json
{
  "operations": {
    "worldbook": {
      "desired_entries": [
        { "name": "殷冬雪", "content": "角色档案的完整文本内容" }
      ],
      "remove_entries": []
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "fallback_entries": ["殷冬雪"]
    }
  }
}
```

就这么简单。

## 三、每个字段怎么写？

### desired_entries（要写入的条目）

每个条目只需要 **name** 和 **content**：

```json
{ "name": "小雪", "content": "小雪当前心情不错，正在公园散步。" }
```

- **name**：条目名称，直接写裸名即可（如 `"小雪"`，不要写 `"EW/Dyn/小雪"`）
- **content**：条目的完整文本内容。注意是**完整覆盖**，不是增量补丁

**条目命名建议**：
- 角色档案：直接用角色名，如 `"殷冬雪"`
- 角色记忆：`"殷冬雪_记忆"`
- 时间线：`"主时间线"`
- 名称每次必须保持一致，不能随意变动

### controller_model（控制条目加载）

这是关键：**光写入条目并不会让 AI 看到它**。你必须在 `controller_model` 里告诉插件"这些条目应该被加载到对话上下文中"。

#### 基础用法：`fallback_entries`

最简单的写法——把所有需要加载的条目名列在 `fallback_entries` 里：

```json
{
  "template_id": "entry_selector_v1",
  "fallback_entries": ["小雪", "主时间线", "小雪_记忆"]
}
```

如果你不提供 `controller_model`，所有条目虽然写进了世界书，但**不会出现在 AI 的对话里**，等于白写。

#### 条件加载：`variables` + `rules`

根据酒馆变量的值，决定加载哪些条目：

```json
{
  "template_id": "entry_selector_v1",
  "variables": [
    { "name": "phase", "path": "story_phase", "default": "early" }
  ],
  "rules": [
    { "when": "phase === 'late'", "include_entries": ["小雪_觉醒", "终章时间线"] }
  ],
  "fallback_entries": ["小雪", "主时间线"]
}
```

逻辑：按 rules 顺序匹配，命中则加载对应条目，全不命中则走 fallback_entries。

#### 角色检测：`char_detection`

自动检测对话中出现的角色并加载对应条目，无需手动列举：

```json
{
  "template_id": "entry_selector_v1",
  "char_detection": {
    "alias_map": {
      "殷冬雪": "殷冬雪", "冬雪": "殷冬雪",
      "卫疏影": "卫疏影", "疏影": "卫疏影"
    },
    "scene_var": "stat_data.在场人物",
    "scan_messages": 1,
    "entry_patterns": ["{name}", "{name}_记忆"]
  },
  "fallback_entries": ["主时间线"]
}
```

| 子字段 | 说明 |
| --- | --- |
| `alias_map` | 别名→正式名映射，正式名也要映射自己 |
| `scene_var` | 可选，酒馆变量路径（值为对象，键名=角色名） |
| `scan_messages` | 扫描最近 N 轮用户+AI消息中出现的角色名，默认 1 |
| `entry_patterns` | 加载模式，`{name}` 替换为检测到的角色名，默认 `["{name}"]` |

#### 通用列表循环：`for_each`

从酒馆变量数组读取列表，为每个元素加载对应条目：

```json
{
  "template_id": "entry_selector_v1",
  "for_each": [
    { "list_var": "active_quests", "entry_prefix": "任务_" }
  ],
  "fallback_entries": []
}
```

比如变量 `active_quests = ["主线1", "支线2"]`，会自动加载 `EW/Dyn/任务_主线1` 和 `EW/Dyn/任务_支线2`。

#### 其他进阶字段

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| `decorators` | 给 Controller 条目添加装饰器 | `["@@preprocessing"]` |
| `skip_floor_zero` | 第一条消息时跳过整个控制器 | `true` |
| `set_variables` | 设置酒馆变量 | `[{ "key": "level", "value": 3, "scope": "local" }]` |
| `activate_entries` | 激活非 EW 条目（不加前缀） | `[{ "entry": "通用设定" }]` |
| `inject_text` | 直接注入文本到上下文 | `["注意：当前是第三章"]` |

> 所有字段都是**可选的**，大多数场景只需要 `fallback_entries` 就够了。

### remove_entries（要删除的条目）

当某个条目不需要了（比如 NPC 离场、约定已兑现），放进这里：

```json
"remove_entries": [{ "name": "过期的NPC" }]
```

通常情况下传空数组 `[]` 就行。

## 四、工作流提示词怎么写？

你需要在工作流的提示词编排里告诉 AI 两件事：
1. **做什么**（任务指令）
2. **怎么输出**（格式要求）

### 建议的提示词结构

| 顺序 | 内容 | 作用 |
|---|---|---|
| 靠前 | 角色定义、聊天记录等系统标记 | 给 AI 提供上下文 |
| 中间 | 任务指令（你希望 AI 做什么） | 具体的更新逻辑 |
| 靠后 | 格式约束（JSON 输出要求） | 确保 AI 按规定格式输出 |
| 最后 | COT 引导（可选） | 让 AI 先思考再输出 |

> 格式约束放在越后面效果越好，因为模型更注重靠后的指令。

## 五、应用场景模板

### 模板 A：角色状态更新

适用于根据剧情推进，动态更新角色的当前状态、态度、外观等。

```text
你是一个角色状态更新引擎。阅读最新对话，更新角色和 NPC 的动态信息。

你需要做的事：
1. 读取最近对话，识别角色的状态变化
2. 对比已有条目，只更新有变化的
3. 输出 JSON，在 fallback_entries 中列出所有应生效的条目名

更新维度（按需选取）：
- 情绪/心理状态
- 对用户的态度变化
- 外观/位置变化
- NPC 关系变化

条目命名：直接用角色名（如 "小雪"），插件会自动加 EW/Dyn/ 前缀。
每个条目的 content 写当前状态，用第三人称、简洁的描述性文本。
只写变化的部分，没变化的角色不要输出。

严格只输出 JSON 对象。
```

### 模板 B：增强记忆

适用于提取对话中的关键事件、承诺、信息，写入长期记忆。

```text
你是一个记忆管理引擎。从最新对话中提取值得长期记住的信息。

你需要做的事：
1. 阅读最新对话，识别值得记忆的信息
2. 对比已有记忆，避免重复，合并或更新过时的
3. 输出 JSON，在 fallback_entries 中列出所有应生效的记忆条目名

值得记忆的信息：
- 关键事件（约会、争吵、告白等）
- 承诺/约定（"下周一起看电影"）
- 用户个人信息（喜好、经历）
- 关系转折点

条目命名建议：
- "记忆_事件_简短描述"（如 "记忆_事件_公园散步"）
- "记忆_约定_简短描述"
- "记忆_用户_简短描述"

每条记忆 1~3 句话，客观概括。过时的记忆放进 remove_entries。
总记忆条目建议控制在 10~20 条。

严格只输出 JSON 对象。
```

### 模板 C：人设 + 记忆一体化

只用一个工作流同时处理角色状态和记忆管理。

```text
你同时负责两项工作：更新角色状态 + 管理记忆。

角色状态更新：
- 条目名用角色名（如 "小雪"）
- 只写当前状态的变化部分

增强记忆管理：
- 条目名用 "记忆_分类_描述" 格式
- 每条 1~3 句话，客观概括
- 旧记忆过时则放入 remove_entries

关键：在 controller_model.fallback_entries 中列出所有应生效的条目名。

严格只输出 JSON 对象。
```

## 六、COT 思考引导（可选）

配合 `<thinking>` 标签使用。让 AI 先想清楚再输出 JSON，提高输出质量。

```text
先在 <thinking> 标签内完成分析：
1. 最近对话发生了什么重要的事？
2. 哪些角色的状态需要更新？
3. 有没有值得记忆的新信息？
4. 已有的条目是否需要修改或删除？

然后在 </thinking> 之后输出最终的 JSON 对象。
```

> 使用 thinking 标签时，需要在工作流的**响应后处理**里配置移除正则（如 `<thinking>[\s\S]*?</thinking>`），插件会自动剥离思考内容再提取 JSON。

### 示例 1：角色状态更新

场景：小雪因为用户迟到而生气，店长出场解围。

```json
{
  "operations": {
    "worldbook": {
      "desired_entries": [
        {
          "name": "小雪",
          "content": "小雪当前心情不好，因为用户迟到了20分钟。表面说没关系，实际上有些生气。坐在咖啡厅靠窗位置，手里拿着凉了的拿铁。"
        },
        {
          "name": "NPC_店长",
          "content": "咖啡厅店长，中年女性，热心。注意到小雪等了很久，主动送了块蛋糕。对迟到的用户态度友善但带打量。"
        }
      ],
      "remove_entries": []
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "fallback_entries": ["小雪", "NPC_店长"]
    }
  }
}
```

场景：用户提到下周生日，和角色约好了周六去水族馆。

```json
{
  "operations": {
    "worldbook": {
      "desired_entries": [
        {
          "name": "记忆_用户_生日",
          "content": "用户提到自己下周过生日，这是第一次提到生日信息。"

      "remove_entries": []
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "fallback_entries": ["记忆_用户_生日", "记忆_约定_水族馆"]

  }
}
```

### 示例 3：一体化更新

场景：小雪和用户一起淋雨回家，关系变得更亲近。

```json
{
  "operations": {
    "worldbook": {
      "desired_entries": [
        {
          "name": "小雪",
          "content": "小雪和用户一起淋雨跑回家，裹着毯子坐在沙发上。虽然冷但心情好，觉得这种经历拉近了距离。态度从客气转为自然亲近。"

      "template_id": "entry_selector_v1",
      "fallback_entries": ["小雪", "记忆_事件_一起淋雨"]
    }
  }
}
```

## 八、常见问题

**Q：条目写进去了但 AI 看不到？**
A：检查是否提供了 `controller_model`，并在 `fallback_entries` 中列出了条目名。

**Q：条目名需要加 `EW/Dyn/` 前缀吗？**
A：不需要，插件自动添加。你只写裸名就行。

**Q：`enabled` 字段需要写吗？**
A：不需要，插件自动处理。Dyn 条目始终设为禁用状态（由控制器按需加载）。

**Q：一次可以更新多少条目？**
A：没有硬性限制，但建议控制在合理范围内。只更新有变化的条目。

**Q：`controller_model` 的 `variables` 和 `rules` 什么时候用？**
A：当你希望根据酒馆变量条件性地加载不同条目时才用。大多数场景用 `fallback_entries` 就够了。

**Q：`char_detection` 和 `for_each` 有什么区别？**
A：`char_detection` 专门用于角色检测——它扫描消息文本和变量，找到在场角色后加载对应条目。`for_each` 是通用循环——从酒馆变量数组读取列表，按前缀/后缀拼接条目名加载，适合任务、地点等非角色类条目。

**Q：`decorators` 中的 `@@preprocessing` 有什么用？**
A：让 Controller 条目在世界书处理之前执行，可以动态激活其他绿灯条目。一般不需要。
